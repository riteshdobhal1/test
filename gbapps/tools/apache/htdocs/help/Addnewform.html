Add new form

  1. Follow the table as you walk through each of the options in the form. 

Item

What you should do

Label

Provide a descriptive name to identify rule. 

Choose a name or Label for your alert. The name will be returned with any alert that is triggered.

You can enter label in the following formats:

  • Any text
  • {sec.attributes}
  • JSON.  Click here to know more about JSON support.

Description

Enter a description. 

It could be a message that appropriately describes a rule so that you can remember why you wanted this rule.

Category

Choose an appropriate category. If you want to setup a new category, then add a new category. 

Select a category to which the rule will cater, or add a new category to which the rule must belong. For details, see the section “To add a new category”.

Author

This is the name of the user who is creating the Rule. This is for your peers to know, who created the rule, so that they can contact the creator if needed.  

Severity

Set the severity for a given rule. Setting the right severity is important as it helps the users who view the alert to understand its severity or importance. All rules that are set to the highest severity are displayed at on top of the Rules List page. 

Priority

Set the priority for a given rule. Setting the right priority is important as it helps the users who view the alert to understand its priority or importance.

Scope

Option

Rule Criteria

Trigger Criteria

Row

When the condition of the rule is within a single line in the log

When you want to trigger the rule on every line if it meets the condition

File

When the condition of the rule is across multiple lines in the log, but those lines are within a given file.

You should use multiple Count function for multi sections. For example,

count(section1.name="name1") > 0 AND/OR count(section2.name="name2") > 1

When you want to trigger the rule, only once per file, even if it occurs more than once

Bundle

When the condition of the rule is across multiple lines in the log and those lines are across different files within the bundle

You should use multiple Count function for multi sections. For example,

count(section1.name="name1") > 0 AND/OR count(section2.name="name2") > 1

When you want to trigger the rule only once per bundle, even if it occurs more than once

Action

What to do when an alert is triggered. 

Persist:  When the Action is set to Persist, the alert will be persisted in the database for the user to view the alerts through an application/dashboard. 

Persist and e-mail:  If the Action is set to Persist and e-mail, then along with persistence, an E-Mail will also be sent based on the template attached to this rule and the subscription list.  Click here to know more about rule subscription. 

Persist and Call API:  If the action is set to Persist and Call API, the alerts persist in the database.  Also, the API (associated with API template) is called and the payload data is also passed to the API.

Email Template

Select the mail template or create a new template by clicking the Manage Templates button. See the “Creating an Email Template” section on how to create the content for setting up the alert message. 

API Template

List of all the API templates created.  Click here to select an API template from the drop down list.

Max Alerts (email)

By default it is 100. This is the maximum number of email messages per log bundle per rule that is sent whenever the condition (logic) is met. Therefore, there can a maximum 100 email messages if even the logic condition exceeds 100 times for a rule in a given log bundle. This is done to limit E-mail blast. If a rule is not set correctly, it will end up sending thousands of mails. 

Max Alert (View)

This is a number that can be set per rule, which decides how many alerts need to be shown in the dashboard and also when exported. The rule metadata just stored this information per rule and it is the responsibility of the App layer to make use of it.

KB Link

Link to an internal or external knowledge base article to help peers get more information on the cause or effect of the rule. This field can be used by the author of the rule to provide more details about the problem and its resolution for which the rule was set. 

Recommendation

Enter the remedial action your peers should take to rectify the system issues at hand when they see this alert.  You can enter recommendation in the following formats:


  • Link to external source
  • Any text
  • {sec.attributes} / {evt.attributes}
  • JSON.  Click here to know more about JSON support.

Justification 

This attribute allows Rule creators to explain the justification for the triggered alert. 

You can enter justification in the following formats:

  • Link to external source
  • Any Text
  • {sec.attributes} / {evt.attributes}
  • JSON.  Click here to know more about JSON support.

Logic

Use the Operators, Functions, and Attributes list items to create conditions that need to assess in a log file.

For instance: “{Show whitelist-dbcpsec-status.Unapproved entries} > 0 OR {Show whitelist-dbcpsec-status.Certified hold entries} > 0 OR {Show whitelist-dbcpsec-status.Revoked entries} > 0”

In the above logic construction, the green entries are attributes items that are part of the log sections. Rest of the computation is part of the validation logic to compute the values of the attributes against other entries in the log sections. These can be accomplished using simple mathematical operators and functions, available in the left panel. 

Text

This field is used to define the format and the fields that you would like to see in the alert, when this rule is triggered. For instance, whenever the condition is met, you may want to describe the actual problem in the alert, you might want to pick other attributes and their values that can be used by troubleshooters to identify the condition in the machine.  

For instance, when the above logic is executed. This can be the possible alert condition to trigger:

Show whitelist-dbcpsec status :: CPSEC: APs are not coming up because of either Unapproved entries or Certified hold entries or Revoked entries have non Zero value : Current State is - Unapproved entries = {Show whitelist-dbcpsec-status.Unapproved entries}, Certified hold entries = {Show whitelist-dbcpsec-status.Certified hold entries}, Revoked entries = {Show whitelist-dbcpsec-status.Revoked entries}

Note: The text inside {} is the extracted column, whose value will be replaced in the text. 

This text content with the values extracted from the troubling log is displayed as the alert condition in the Rules List and the email message alert.

 You can enter text in the following formats:

  • Any Text
  • {sec.attributes}
  • JSON.  Click here to know more about JSON support.

Once you have completed all of the mandatory entries, check again for the Logic and Text field for its appropriateness and accuracy before you click the Submit.  Once you click Submit, you will get the “Rule added successfully as draft. ID is “Rule_xxxxxxxx”” messaged displayed on the top of the page. 


Now, the rule needs to be tested for its accuracy. This can be completed in the next stage of the Rules creation stages.
