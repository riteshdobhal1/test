About Glassbeam Rules & Alerts

Glassbeam Rules and Alerts is a powerful rules engine that allows complex rules to be setup for proactive action. Rules are conditions.  Alerts are triggered when the conditions are met.  Modern architecture and design enables rules to be setup from data across multiple sources. Rules are evaluated in real time while parsing data bundles. Rules can be setup at a log section level, file level or bundle (multiple log files sent as .zip, .tar etc.) level.

A tech support log bundle from a device contains many files containing multiple sections, events-related messages, and so on. One of the biggest use cases for mining machine data from devices is to proactively identify conditions in log that requires someone to be notified. Let’s take some examples to understand this:

  1. Let’s assume that we are receiving data about pressure, temperature and volume of a liquid flowing into a machine. While plotting these values and visualizing the trend over time is meaningful and required for analysis, triggering an alert when one of the values crossed threshold is required for proactive monitoring. 
  2. Let’s assume a support engineer solves a specific problem for a customer after spending days to troubleshoot. The engineer now has a clear idea of which sections in a log file or which log sources to look at to identify this problem. This engineer would like to teach the system, how to identify this problem, so that going forward any log uploaded with this anomalous value is automatically identified.
  3. A product company just released a new version of their software and they realized that the new version has a major bug. The bug manifests itself only in specific conditions and there is a way for Engineering to find out early signs of this issue. Engineering wants to setup a rule to identify this condition for every log file received from customers, so that they can proactively address the issue with the customer.
  4. The sales head of a product company wants to understand when would be the right time to upsell into an account. The Sales team comes up with a set of conditions that if met, they know that the customer would be looking for an upgrade. They would like to monitor data from every device continuously and alert the inside sales team, if any account meets the criteria.

Each of the examples above shows different use cases where a Rules and Alerts engine would help. Glassbeam’s SCALAR platform and SPL helps extract information from machine logs and Glassbeam’s Rules and Alerts engine is integrated into its SCALAR platform, which ensures that every section or log file parsed using SPL, is available for the Rules Engine to set a rule on.
